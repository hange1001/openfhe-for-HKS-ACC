  ============================================================================
  二次优化：针对综合结果恶化的根因修复
  ============================================================================

  问题背景：
  上述修改方案实施后，综合结果反而变差——综合时间剧增、资源爆炸、流水线 II 违规。
  经分析，根因集中在三个方面：
  1. Twiddle Factor 降维为 1D 后产生严重 Bank 冲突
  2. complete / cyclic factor 不匹配导致 MUX 爆炸
  3. 函数调用边界 partition 不一致导致 Crossbar 重排

  以下是二次修复的完整方案（已全部实施到代码中）。

  ---
  核心设计决策：define.h 新增 PE_PARALLEL 全局常量

  // define.h 新增（位于 BU_NUM 定义之后）
  static const int PE_PARALLEL = 8;

  含义：所有 ARRAY_PARTITION cyclic factor、UNROLL factor、Twiddle 副本数
  统一引用此常量。未来调整并行度只需修改一处。

  ---
  FIX-A：消除 complete / MUX 爆炸（P0 级）

  问题：ntt_kernel.cpp 中 buf_A/buf_B 用了 complete dim=2（展开 64 列）
  + BIND_STORAGE ram_2p bram，产生 64:1 MUX 和数百个 BRAM 实例。
  top.cpp 中 poly_buffer 用了 cyclic factor=SQRT(=64)，等效 complete。

  修复原则：全链路统一为 cyclic factor=PE_PARALLEL，与 UNROLL factor 对齐。

  FIX-B：Twiddle Factor 恢复多副本 + URAM 映射（P1 级）

  问题：降维为 1D [RING_DIM] 后，UNROLL factor=8 的 8 个 PE 在早期 NTT Stage
  会计算出完全相同的 TwiddleIndex（公式中 l >> 11 对 l=0..7 全为 0），

  - 每张表: PE_PARALLEL × RING_DIM × 8B = 8 × 4096 × 8 = 256 KB
  - MAX_LIMBS 个 limb × 2 张表(NTT+INTT) = 8 × 256KB × 2 = 4 MB
  - U55C URAM 总量 ≈ 34 MB → 占用 ~12%，完全可承受

  ---
  FIX-C：MultMod 延迟锁定（P2 级）

  问题：bconv.cpp 脉动阵列中 MULTMOD_LAT=4 硬编码了 MultMod 的延迟，
  用于移位寄存器补偿。如果 HLS 在不同综合策略下改变 MultMod 实际延迟，
  脉动阵列的时序对齐会被破坏，产生错误结果。

  修复：arithmetic.cpp 中 MultMod 函数显式锁定流水线深度

  void MultMod( ... ) {
      #pragma HLS INLINE off
      #pragma HLS PIPELINE II=1
      #pragma HLS LATENCY min=4 max=4  // 锁定，严格匹配 MULTMOD_LAT=4
      // ... 计算逻辑不变 ...
  }

  效果：如果 DSP 资源不足或优化策略变化导致延迟不能为 4，
  HLS 工具会直接报错而非静默生成错误硬件。
